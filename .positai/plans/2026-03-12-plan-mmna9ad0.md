# Diagnosing and Fixing the ~10% LAPJV Benchmark Regression

## Root Cause Analysis

### What changed in the LAPJV compilation path?

`lap.cpp` includes `tree_distances.h`. The source of `lap.cpp` itself is **identical** across all three branches (main, posit-optim-2, posit-optim-3). The only changes affecting its compilation come through the header.

**PR #178 (posit-optim-2 vs main):** The sole change to `tree_distances.h` is a 5-line refactor of `one_overlap()` ŌĆö a function that `lap.cpp` never calls. CostMatrix, LapScratch, and all LAP-related code are byte-for-byte identical.

**PR #180 (posit-optim-3 vs posit-optim-2):** CostMatrix gains a default constructor, `resize()`, and `const` is removed from `dim_`/`dim8_`. LapScratch gains two `CostMatrix` pool members.

### Why does an unrelated header change cause a 10% LAP regression?

This is an **instruction alignment artifact**. The x86 instruction decoder fetches aligned 16-byte or 32-byte blocks. When the compiler emits code for `lap.cpp` at `-O3`, the layout of hot loops (Dijkstra inner loop, `findRowSubmin`) is sensitive to their alignment within instruction cache lines. Changes anywhere in the header ŌĆö even to functions never called ŌĆö can shift the code layout enough to move a hot loop across an alignment boundary, degrading branch prediction or instruction fetch throughput.

Key evidence:
- `lap.cpp` source is unchanged; only the included header changed
- The regression is consistent across all matrix sizes (40, 400, 2000) ŌĆö ruling out algorithmic causes
- The regression is reproducible across 3+ independent CI runs
- For PR #178, the CostMatrix/LapScratch structs are **identical** to main ŌĆö only an unrelated `one_overlap` inline function changed

For PR #180, there's also a secondary mechanism: removing `const` from `dim_`/`dim8_` reduces the compiler's ability to optimize `lap()`, since it takes `CostMatrix&` by non-const reference and can no longer assume these fields are immutable. However, since PR #178 already shows the full ~10% regression with `const` intact, this appears to be a minor contributor at most.

## Fix: Header Split

The fix requires **no code duplication** of the LAP itself. We compile `lap.cpp` against a minimal header so that future changes to tree-distance scoring functions can't affect LAP code generation.

### Current structure of `tree_distances.h` (620 lines)

| Lines | Content | Used by LAP? |
|-------|---------|-------------|
| 1ŌĆō31 | Types (`cost`, `lap_dim`, etc.), constants (`BIG`, `BLOCK_SIZE`), `#include`s | Yes |
| 33ŌĆō401 | `CostMatrix` class (with `findRowSubmin`, `findColMin`, `row`) | Yes |
| 403ŌĆō461 | `cost_matrix` alias, `LapScratch`, `lap()` declarations | Yes |
| 463ŌĆō620 | `namespace TreeDist { add_ic_element, one_overlap, spi_overlap, ... }` | **No** |

`lap.cpp` uses exactly 3 CostMatrix methods (`findColMin`, `row`, `findRowSubmin`) and zero functions from `namespace TreeDist`. It also doesn't use `lg2[]`, `splitbit`, `ALL_ONES`, or any tree-specific globals ŌĆö only `BIG`, `BLOCK_SIZE`, and `ROUND_PRECISION`.

### Implementation steps

**Step 1: Create `src/lap.h`** ŌĆö extract lines 1ŌĆō461 from `tree_distances.h`:
- `#include`s needed by CostMatrix (`<TreeTools/SplitList.h>`, `<Rcpp/Lightest>`, `<limits>`, `<vector>`, `"ints.h"`)
- Type aliases (`cost`, `lap_dim`, etc.)
- Constants (`BIG`, `ROUND_PRECISION`, `BLOCK_SIZE`)
- `CostMatrix` class (full definition including all member functions)
- `cost_matrix` alias
- `LapScratch` struct
- `lap()` function declarations (both overloads)
- Own include guard (`_TREEDIST_LAP_H`)

**Step 2: Refactor `tree_distances.h`**:
- Replace the extracted content with `#include "lap.h"`
- Keep tree-specific globals (`ALL_ONES`, `lg2[]`, `lg2_rooted`, etc.)
- Keep `namespace TreeDist { ... }` (scoring functions)
- Keep its own include guard

**Step 3: Update `lap.cpp`**:
- Change `#include "tree_distances.h"` ŌåÆ `#include "lap.h"`

**Step 4: No changes to other `.cpp` files** ŌĆö they continue to `#include "tree_distances.h"` which now pulls in `lap.h` + scoring functions. Files affected: `pairwise_distances.cpp`, `tree_distances.cpp`, `tree_distance_functions.cpp`, `day_1985.cpp`, `nni_distance.cpp`, `hmi.cpp`, `hpart.cpp`.

**Step 5: Verify** ŌĆö run the A/B benchmark via `compare-ab.R` to confirm LAPJV performance returns to baseline while tree distance improvements are preserved.

### Why this works

- `lap.cpp` no longer sees the `namespace TreeDist` inline functions that were causing alignment shifts
- Future changes to scoring functions (`one_overlap`, `add_ic_element`, `spi_overlap`) only affect `tree_distances.h`, which `lap.cpp` no longer includes
- Zero code duplication ŌĆö `lap.h` is a strict subset extracted from the existing header
- All other translation units see identical content via `#include "tree_distances.h"`

### Maintenance burden

Minimal. The split is along a natural boundary:
- `lap.h` = stable LAP infrastructure (types, CostMatrix, LapScratch) ŌĆö changes rarely
- `tree_distances.h` = actively-developed scoring functions ŌĆö changes frequently

When CostMatrix or LapScratch need modification, the change goes in `lap.h` and is automatically picked up by both `lap.cpp` and `tree_distances.h` consumers. No duplication, no divergence risk.

### What this does NOT fix

The CostMatrix `const` removal (dim_/dim8_ no longer const) is in `lap.h` and still visible to `lap.cpp`. This is a secondary effect ŌĆö PR #178 shows the full ~10% regression was present even with `const` intact, so the alignment artifact from scoring functions is the primary cause. If the `const` removal turns out to matter post-split, we can address it separately (e.g. a `ResizableCostMatrix` subclass for the batch path only).
